# 들어가며

제가 중학생이던 당시 특수교육과에 재학 중이던 사촌 언니는 시각장애인 친구들이 있었습니다. 같이 볼링도 치고 카카오톡도 한다는 이야기를 듣고 많이 놀랐습니다. 저는 장애를 가지고 있기에 많은 활동에 제약이 있을 것이라 막연히 편견을 가지고 있었기 때문입니다. 카카오톡을 어떻게 할 수 있는지 궁금했고, 언니는 휴대폰에서 텍스트를 읽어준다고 했습니다. 그때 그들도 똑같이 스포츠를 즐기고 사람들과 소통하면서 살아간다는 것을 크게 알게 되었고, 당연히 이용하지 못할 것이라 생각했던 제가 부끄러웠습니다. 시간이 흘러 저는 많은 사람들에게 좋은 의미를 줄 수 있는 서비스를 만드는 사람이 되고 싶다는 생각으로 기술을 배우게 되었습니다. 그때의 일이 묘하게 부채감으로 남은 저는 스크린리더를 개선하는 작업에 더욱 의미를 가지고 임할 수 있었습니다.

‘땅콩’은 그룹이 함께 대화를 나누며 게임을 즐길 수 있는 서비스를 제공합니다. 땅콩 팀은 “사용자에게 즐거운 대화를 제공한다.”는 핵심 가치에 따라 사용자 중심으로 만들어 왔습니다.

하지만, 현재 서비스는 접근성에서 몇 가지 제약이 있습니다. 그중에서도 특히 스크린 리더를 사용하는 사용자들이 중요한 정보나 인터페이스를 충분히 인식하지 못하는 문제가 발견되었습니다. 이러한 문제를 해결하기 위해 우리는 스크린 리더 접근성을 우선적으로 개선하기로 결정했습니다. 이를 통해 시각적 피드백에 의존하지 않는 사용자들도 동일한 게임 경험과 즐거운 대화를 누릴 수 있도록 목표를 설정했습니다.

# 스크린 리더

## 스크린 리더란?

스크린 리더란 무엇인가?
스크린 리더(Screen Reader)는 시각적 콘텐츠를 인식하지 못하는 사용자들이 웹사이트나 애플리케이션을 사용할 수 있도록 돕는 보조 기술입니다. 텍스트, 이미지, 링크, 버튼 등의 요소를 음성으로 변환하거나, 점자 디스플레이에 출력하여 사용자에게 정보를 전달합니다. 주요 스크린 리더로는 Windows의 JAWS, NVDA, macOS의 VoiceOver, iOS의 VoiceOver, Android의 TalkBack 등이 있습니다.

스크린 리더는 웹 페이지의 HTML 구조와 시멘틱 태그를 기반으로 정보를 해석합니다. 따라서, 올바르게 구성된 HTML 태그와 접근성 속성(ARIA)을 사용해야 스크린 리더가 정확하게 읽어낼 수 있습니다.

## 스크린 리더의 주요 특징

1. 순차적 읽기: 스크린 리더는 웹 페이지의 내용을 순차적으로 읽어줍니다. 따라서 페이지가 논리적으로 구성되고, 시멘틱 태그가 제대로 사용되어야 사용자가 필요한 정보를 쉽게 파악할 수 있습니다.

2. 시멘틱 HTML의 중요성: `<header>`, `<nav>`, `<main>`, `<footer>`와 같은 시멘틱 태그는 페이지의 구조를 명확하게 구분해 줍니다. 스크린 리더는 이러한 태그를 인식해 페이지 내 섹션별로 안내할 수 있습니다.

3. 대체 텍스트 제공: 이미지와 같은 시각적 요소에는 alt 속성을 사용하여 설명을 제공해야 합니다. 중요한 정보를 담고 있는 이미지라면, 그 내용을 간단하고 명확하게 설명하는 대체 텍스트가 필요합니다.

4. ARIA 속성 활용: 스크린 리더는 ARIA 속성을 통해 더 복잡한 동적 콘텐츠나 상태 변화를 이해할 수 있습니다. 예를 들어, 버튼의 상태나 팝업 대화 상자의 역할을 명확히 전달하는 데 유용합니다.

## 스크린 리더 사용 시 주의 사항

1. 시각적 정보에만 의존하지 않기: 중요한 정보는 텍스트로도 제공되어야 합니다. 예를 들어, 버튼의 모양이나 색상만으로 의미를 전달하기보다는 스크린 리더가 읽을 수 있는 텍스트로 설명을 추가하는 것이 필요합니다.

2. 포커스 관리: 스크린 리더 사용자들은 주로 키보드로 탐색합니다. 중요한 요소가 포커스를 제대로 받을 수 있도록 관리하는 것이 중요하며, 포커스가 엉뚱한 곳으로 이동하지 않도록 주의해야 합니다.

3. 의미 있는 대체 텍스트 작성: 구체적이고 유의미한 대체 텍스트를 제공해야 합니다. 단순히 "버튼"이라고 쓰기보다는 그 버튼의 기능을 설명하는 것이 좋습니다. 예를 들어, "제출 버튼"이나 "장바구니에 추가"와 같이 구체적으로 적어야 합니다.

4. ARIA 속성의 남용 금지: ARIA 속성은 필요한 곳에만 사용해야 합니다. 남용하면 오히려 혼란을 줄 수 있으며, 잘못된 사용은 스크린 리더가 정보를 제대로 전달하지 못하게 만들 수 있습니다.

# 개선 계획

## 공통

1. 페이지 포커스 문제 해결: 페이지가 로드되거나 새로고침될 때 포커스가 자동으로 상위 요소로 이동하지 않고, 사용자가 현재 페이지의 중요한 요소나 첫 번째 인터랙션 가능한 요소로 바로 접근할 수 있도록 포커스 흐름을 개선합니다. 이를 통해 사용자는 페이지의 가장 중요한 정보를 놓치지 않게 됩니다.

2. 모달 포커스 관리:

- 모달 내로 포커스 이동: 모달이 열릴 때 포커스가 자동으로 모달 내로 이동하여 사용자가 모달 내의 인터랙션 요소를 쉽게 탐색할 수 있도록 합니다. 이를 통해 모달의 모든 콘텐츠가 스크린 리더를 통해 명확하게 안내될 수 있습니다.
- 모달 종료 시 이전 포커스로 복귀: 모달이 닫히면 포커스가 모달을 열기 직전의 위치로 돌아가도록 하여, 사용자가 이전에 인터랙션하던 위치에서 다시 작업을 이어갈 수 있게 개선합니다. 이를 통해 모달을 사용한 후 사용자가 인터페이스를 쉽게 이어서 탐색할 수 있도록 합니다.

3. 뚝뚝 끊겨 읽지 않고 한번에 읽어야 하는 요소는 한번에 읽도록 개선합니다. ex) 80, % -> 80%

4. 필요하지 않는 부분은 읽지 않도록 개선합니다.

## 닉네임 설정 페이지

1. 닉네임 필드에 대한 명확한 안내: 스크린 리더가 닉네임 입력 필드를 명확하게 안내하도록 개선합니다.
2. 닉네임 글자수 제한에 대한 안내: 사용자가 입력 중인 닉네임과 남은 글자 수를 쉽게 파악할 수 있도록 개선합니다.

## 게임 대기 페이지

1. 방 설정 정보 안내: 현재 게임에 어떤 설정이 되어 있는지 명확히 설명하여, 스크린 리더 사용자가 방의 설정을 이해할 수 있도록 개선합니다.

2. 왕관 아이콘 설명: 왕관 아이콘이 방장을 의미한다는 사실을 명확히 설명하여, 스크린 리더 사용자가 방장의 역할을 이해할 수 있도록 개선합니다.

3. 참여 인원 안내: 현재 게임에 참여 중인 인원의 수를 정확하게 인식할 수 있도록 실시간으로 인원 변화를 안내합니다.

4. 대기 인원 안내: 현재 게임에 참여 중인 참여자들의 프로필과 닉네임이 안내 없이 나열되어 있으므로 이를 명확히 설명하여, 스크린 리더 사용자가 대기 중인 인원임을 이해할 수 있도록 합니다.

### 게임 대기 페이지 - 초대하기 모달

1. 큐알 코드 안내 개선: 큐알 코드 이미지를 명확하게 설명하여, 스크린 리더 사용자가 큐알 코드의 용도를 인식할 수 있도록 개선할 계획입니다.

2. 링크 복사 버튼의 안내 개선: 링크 복사 버튼을 클릭했을 때, 해당 작업이 성공적으로 이루어졌음을 사용자에게 명확히 전달하여 복사 완료 상태를 인식할 수 있도록 개선합니다.

### 게임 대기 페이지 - 방 설정 모달

1. 드롭다운 메뉴의 접근성 강화: 드롭다운이 열리고 닫히는 상태를 스크린 리더가 명확하게 인식할 수 있도록 개선하여, 사용자가 선택한 항목을 쉽게 알 수 있도록 합니다.

2. 타이머 설정 버튼 설명: 타이머 설정 버튼의 역할과 기능이 명확히 설명되도록 하여, 사용자가 설정된 시간을 정확히 인식하고 조정할 수 있게 할 수 있도록 합니다.

## 게임 페이지

1. 선택된 옵션에 대한 피드백 제공: 스크린 리더가 사용자가 선택한 답변을 명확히 인식하고, 선택이 성공적으로 반영되었음을 알 수 있도록 개선합니다.

2. 타이머 진행 안내: 남은 시간을 실시간으로 음성으로 안내하여, 스크린 리더 사용자가 제한 시간을 인식하고 선택할 수 있도록 개선합니다.

3. 선택 완료 버튼 개선: 선택지를 누르면 선택되었음을 안내하여, 스크린 리더 사용자가 선택한 선택지를 인식할 수 있도록 개선합니다.

## 라운드 결과 페이지

### 투표 통계 탭

1. 투표 통계 설명 강화: 스크린 리더가 각 선택지에 대한 투표 통계와 비율을 정확히 인식하고, 사용자에게 명확히 안내할 수 있도록 개선합니다.

### 투표 현황 탭

1. 투표 참여 현황에 대한 명확한 안내: 스크린 리더가 각 사용자의 투표 참여 여부와 선택지를 명확하게 인식하고 안내할 수 있도록 개선할니다.

## 매칭 결과 페이지 개선 계획

1. 매칭도 설명 강화: 매칭된 사용자와 그들의 매칭 비율이 구체적으로 설명되도록 하여, 스크린 리더 사용자가 매칭 결과를 명확히 이해할 수 있도록 합니다.

# 개선 과정

## 개선 목표

기능 개발과 버그 수정과 병행하여 작업이 진행되어야 했기 때문에 개선 목표를 처음부터 크게 잡지 않고 1차 개선과 2차 개선으로 나누어서 진행하였습니다. 1차 개선의 목표는 땅콩의 주요 기능이 있는 페이지, 2차 개선의 목표는 나머지 기능이 있는 페이지를 개선하고 최종 목표는 땅콩의 서비스를 스크린리더로 사용할 수 있도록 하는 것으로 설정하였습니다.

## 우선 순위 설정

![alt text](https://raw.githubusercontent.com/woowacourse/woowa-writing/useon/priority.png)
각 목표에 해당 하는 기능을 개선하기 위한 테스크를 4가지로 분류하여 우선 순위를 설정하고 작업을 분배하였습니다. 분류 기준은 다음과 같습니다.

1. 구현이 쉬우면서 개선 효과가 큰 테스크
2. 구현이 어렵지만 개선 효과가 큰 테스크
3. 구현이 쉽지만 개선 효과도 작은 테스크
4. 구현이 어려운데 개선 효과도 작은 테스크

## 개선 방법

프론트엔드 세 명이 동시에 작업을 진행해야 했기 때문에 최대한 동일한 방식으로 개선이 되도록 기준을 설정하였습니다. 기준은 다음과 같습니다.

1. 시맨틱 태그를 쓰자

   > 기존의 코드를 많이 바꾸지 않고 의미에 맞는 태그를 사용하는 것부터 시작하기로 했습니다. 시맨틱 태그는 암묵적으로 롤을 가집니다. 예를 들어, 인풋의 타입이 라디오이면 암묵적으로 부여되는 역할은 라디오입니다.

2. ARIA 속성은 꼭 필요한 경우에만 사용한다
   > ARIA Authoring Practices Guide (APG) 문서에 따르면 **No ARIA is better than Bad ARIA** 라고 합니다. 잘못된 aria 사용으로 안하여 오히려 사용성을 떨어뜨릴 수 있으니 신중하게 사용해야 합니다. <br>
   > 출처: https://www.w3.org/WAI/ARIA/apg/practices/read-me-first/
3. 문서를 잘 읽자
   > 어떻게 들리는 것이 더 편한지는 주관적일 수 있기 때문에 아래와 같은 문서들을 참고하여 개선 작업을 진행했습니다.<br>
   > WCAG 2.1 <br>
   > ARIA Authoring Practices Guide (APG) <br>
   > MDN <br>
   > 네이버의 NULI 지침
4. 스크린리더 전용 컴포넌트를 사용하자

   시각적으로 보이지 않는 요소를 A11yOnly 컴포넌트로 관리하기로 결정했습니다.

   ![alt text](https://raw.githubusercontent.com/woowacourse/woowa-writing/useon/styleAttributeOrA11yonly.png)
   두 코드를 비교하여 보았을 때, A11yOnly 컴포넌트를 사용하여 작성한 코드가 더 한눈에 스크린리더 전용으로 작성된 코드임을 알 수 있고 유지 보수를 하기에도 용이하다고 생각했기 때문입니다.

## ✅ 게임 화면

### **문제 상황**

- 게임이 시작되었는데 게임이 시작되었는지 알 수 없었습니다.
    - 게임 화면으로 넘어갔을 때 전체 라운드와 현재 라운드를 안내해야 합니다. 
- 헤더의 텍스트를 끊어서 읽고 불필요한 정보까지 전달합니다.
    - 헤더에서 게임 정보를 알려주도록 개선합니다.
- 옵션이 선택된 옵션인지 알 수 없었습니다.
    - 옵션 중 하나만 선택 가능하고, 선택 가능한 옵션인지 알려주도록 개선합니다.
- 현재 타이머가 몇초 남았는지 알 수 없었습니다.
    - 기준을 세우고, 기준에 따라서 사용자에게 현재 타이머를 알려주도록 개선합니다.

### **개선 결과**

- 게임 화면으로 넘어갔을 때 전체 라운드, 현재 라운드, 질문, 제한 시간을 안내하여 게임을 수월하게 진행하도록 개선했습니다.
- 처음 라운드 시작되었을 때, 절반 지났을 때, 5초 남았을 때로 총 3번으로 나눠서 제한시간을 알려주었습니다. `role=”alert”` 를 사용하여 특정 지점에서 제한 시간을 안내하도록 설정했습니다.
- 옵션이 이미 선택된 경우 `선택됨` 이라고 안내 음성이 나오며, `radio` 와 `radiogroup` 을 사용하여 두 개 중 몇 번째라는 안내 음성을 제공합니다.

## ✅ 공통

### Modal Focus 개선

Modal을 열고 닫을 때 focus가 제대로 된 위치를 찾지 못합니다.

- 문제점: Modal이 mount 되었을 때 focus가 Modal의 내부가 아닌 Modal의 바깥쪽에서 움직입니다.
- 해결: Modal이 mount 되었을 때 useRef 사용하여 Modal에 ref를 넣고 useEffect 내부에서 Modal ref의 focus 함수를 실행시켜 Focus가 Modal로 오게 만들었습니다.

https://github.com/user-attachments/assets/690bfa3c-6e19-4fa9-a6ea-05c2e598f506

- 문제점: Modal이 unmount 될 때 focus 기존에 있었던 위치로 돌아가지 않고 엉뚱한 곳으로 focus가 돌아갑니다.
- 해결: 현재 구현 방식은 Modal이 특정 버튼을 클릭하면 Modal 호출 함수가 실행되고 Modal이 뜨는 방식이었습니다. 따라서 호출 함수에 인자로 “특정 버튼의 ref”를 넣어주어 Modal 에서 unmount 될 때 인자로 전달해준 ref로 focus가 되도록 구현해서 리더기 사용자가 Modal 닫히더라도 계속해서 서비스를 이용할 수 있게 만들었습니다.

https://github.com/user-attachments/assets/731d3186-2338-4fd8-8c80-144efc74e250

### 페이지 로드 시 Header에 focus

- 문제점: 버튼을 눌러 다음 페이지로 넘어갈 경우 다음 페이지에서의 fosuc가 맨 위로 올라오는 것이 아닌 이전 페이지의 버튼 위치에 계속 머물러 있어. 리더기 사용자 입장에서는 당황스러운 상황이었습니다.
- 해결: Header 태그에 ref를 넣고 useEffect 내부에서 매 페이지가 첫 로딩 될 때 마다 ref의 focus 함수를 실행시킵니다.
