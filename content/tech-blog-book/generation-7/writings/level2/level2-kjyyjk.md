레벨 2를 마치는 이 시점에 돌아본 나의 학습 키워드는 `토론`과 `실험`이다.

### 실험
김영한님의 강의를 들어본 사람들이라면 한 번쯤 `백문이불여일타`라는 말을 들어봤을 것이다.  
백 번 듣는 것보다 한 번 코드로 쳐보는 것이 더 낫다는 의미이다.  
기존에는 당연히 그렇겠지~`하고 넘겼지만, 우테코에서 제공하는 학습 테스트를 경험하고 난 뒤로는 그 의미가 깊게 와닿았다.  
학습 테스트 외 개인 학습에서도 비슷한 학습 방식과 효율을 경험하고 싶었다.  
따라서 이론으로 배운 것들을 직접 코드로 작성하고 실험해볼 수 있는 테스트 환경, 소위 말해 실험실을 만들었다.  

우선 실험하고자 하는 환경에 맞게끔 프로덕션 코드를 작성했다. 
이후 실험 상황과 그에 따른 동작 결과를 가정했다.  
마지막으로 그에 맞는 테스트를 작성하고 실행해보며 쿼리문 혹은 테스트 도구를 통해 예상과 결과가 일치하는지 확인했다.  
결과가 일치하면 실험에 대한 해석을 주석으로 작성했고, 일치하지 않았다면 그 원인을 생각해보며 또 다른 실험을 추가로 진행했다.

```
실험 간단 예시

/**
 * merge할때 select 쿼리가 나간다.
 * 왜?
 * 준영속 상태는 식별자를 가지지만 영속성 컨텍스트에서 관리하지 않는 객체.
 * 준영속 상태를 merge하면 식별자로 DB에서 조회해와 영속성 컨텍스트에서 관리?
 * 만약 준영속 상태가 DB와 상태가 다르다면 어떻게 될까? -> 현재 테스트가 그런 환경.
 *
 * 그럼 왜 select 쿼리를 날리는거지? 그냥 준영속 상태를 그대로 영속성 컨텍스트에서 관리하면 되는거아닌가?
 *
 * 정답은 준영속 상태를 영속 상태로 만드는 동작 과정에 있다.
 * 내가 말한대로 준영속 상태를 그대로 영속 상태로 만드는 것이 아닌.
 * 준영속 상태의 식별자를 가지고 DB에서 조회해와 영속성 객체를 새로 만든다.
 * 이후 준영속 객체의 상태를 새로 만든 영속성 객체의 상태로 복사한다.
 * 즉 새로 영속성 객체를 만들고 준영속 상태를 복사하는것이고, 새로 영속성 객체를 만들기 위해 select 쿼리가 발생하는 것.
*/
@Test
void merge() {
    ... 생략 ...
    entityManager.merge(customer); // <-- 다시 영속 상태로 변경
    System.out.println("================================");
    entityManager.flush(); // 영속성 컨텍스트가 관리하는 객체 DB 반영
    System.out.println("================================");
    ... 생략 ...
}
```

실험을 키워드로한 학습 방식은 학습에서 재미를 느끼게 해주었다.  
실험하다 보면 어느새 다른 크루들이 모여들어 결과를 확인하고 같이 흥미로워하는 경험 또한 즐거웠기에, 더 좋은 기억으로 남은 것 같기도 하다.  

### 토론
평소 나는 자기주장이 완고하고 고집이 매우 센 사람이다.  
우테코에 들어오기 이전에는 내 생각만 옳고 내가 내린 선택이 곧 정답인 줄로만 알고 지내왔다.  
이런 생각들은 우테코의 철학과 학습 환경들을 마주하고 난 뒤 무너지기 시작했다.  
이를 계기로 새로운 마음가짐과 함께 새로운 학습 방식을 시도하기로 했다.  
그렇게 시도한 방식 중 한 가지가 바로 다양한 사람들의 의견을 듣고 여러 사고를 접해보는 것이었다.  

미션 진행 중 문제를 마주했을 때 캠퍼스를 돌아다니며 다른 크루들의 문제 해결 방식을 물어보았다.  
다른 방식의 장단점을 파악하는 것이 궁극적인 목표였기에, 그에 대한 내 생각을 전달하기도 해보고 꼬리 질문을 달기도 했다.  

=========================화자의 독백(미완x)=========================  
그렇게 토론하며 의견 주고 받으니 다른 방식에 대해서 이해할 수 있었음.  
기존 내 생각의 장단점도 더 깊게 고려할 수 있었음.  
내 생각의 장점이 곧 다른 방식의 단점. 내 생각의 단점이 곧 다른 방식의 장점이었음.  
모든 것이 트레이드 오프임을 깨닫기도 함.  
====================================================================

이후에는 나누었던 다양한 방식들을 개인 노션에 정리하고 그 속에서 나만의 기준을 다시 세워나갔다.

이러한 학습 방식을 통해서 내 생각만 옳다는 기존의 편협한 시각에서 탈출하고, 더 유연하고 넓게 바라보는 개발자로 한 걸음 나아갈 수 있었다.

### 마치며
앞으로 추가해보고 싶은 학습 방식 키워드가 있다면 바로 `지식 공유`이다.  
직접 경험하며 얻은 나의 지식을 다른 사람에게    공유하는 경험까지 추가된다면 확실히 나의 것이 될 것이라고 기대한다.
